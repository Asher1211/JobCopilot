# JC (Job Copilot) — 面试技术手册

> 用于实习面试技术面准备，覆盖项目全部技术栈的含义、应用场景、选型理由。

---

## 一、项目概述

JC 是一个 AI 驱动的求职陪跑平台，核心功能：上传简历 + JD → LLM 匹配分析 → 条件路由（面试准备/简历优化）→ 模拟面试 → 面经 RAG 检索 → 公司调研。

**一句话描述本质：** 用 LangGraph 编排多个 LLM Agent 节点，通过 RAG 检索面经增强生成质量，SSE 流式输出到前端。

---

## 二、后端框架 — FastAPI

### 是什么
Python 异步 Web 框架，基于 Starlette + Pydantic，性能接近 Node.js，自带 OpenAPI 文档生成。

### 在项目中的应用
- 16 条 API 路由（认证、分析、面试、调研、面经管理、用户配置）
- `async/await` 异步处理所有 LLM 调用，不阻塞请求线程
- `EventSourceResponse` 实现 SSE 流式输出（分析结果逐 token 推到前端）
- Pydantic 模型做请求校验 + 响应序列化

### 为什么选
- **异步原生支持** — 项目大量 LLM 调用是 IO 密集型，async 比同步 Flask/Django 吞吐量高 3-5 倍
- **自动 OpenAPI 文档** — `/docs` 直接看所有接口，调试方便
- **SSE 原生支持** — 流式输出只需 `StreamingResponse` 或第三方 `sse-starlette`，无需 WebSocket 复杂度

## 三、Agent 编排 — LangGraph

### 是什么
LangChain 团队出的有向图框架，用图结构（节点 + 边）组织多个 LLM 调用，支持条件路由和状态管理。本质是"LLM 工作流引擎"。

### 在项目中的应用
整个分析流程是一个有向图：

```
START → parse_resume → match_analysis → route_decision
                                            ├── score ≥ 60 → interview_prep
                                            └── score < 60  → optimize_resume
```

- `parse_resume` — 文档解析节点（python-docx + PyMuPDF）
- `match_analysis` — LLM 评分 + 技能缺口分析
- `route_decision` — 条件分支函数，读 state["route"] 决定走哪个节点
- `interview_prep` — RAG 面经检索 → LLM 生成备考建议
- `optimize_resume` — LLM 生成优化版简历 HTML

### 为什么选
- **天然支持条件分支** — 传统 Chain 是线性的，LangGraph 的 `add_conditional_edges` 让"匹配度高走面试流，低走优化流"用 5 行代码表达
- **状态自动传递** — `AgentState(TypedDict)` 定义所有字段，节点返回值自动合并到 state，下一个节点直接读
- **事件流** — `astream_events` 发射每个节点的开始/结束/token 事件，SSE 直接消费

---

## 四、LLM 调用 — LangChain ChatOpenAI

### 是什么
LangChain 封装的 LLM 调用接口，`ChatOpenAI` 不仅能调 OpenAI，任何 OpenAI 兼容 API（DeepSeek、Groq、Ollama 等）都能用。

### 在项目中的应用
- 用户在前端 Settings 页自配 API Key + Base URL + Model
- `backend/agents/llm.py` 中 `get_llm()` 工厂函数根据用户配置创建 LLM 实例
- 加密存 PostgreSQL，使用 Fernet 对称加密
- 支持所有 OpenAI 兼容 provider，切换只需改 Base URL

### 为什么这样设计
- **降低运营成本** — 服务端不存任何 API Key，用户用自己的
- **灵活切换模型** — 同一套代码，可跑 DeepSeek/GPT-4o/Groq/Ollama
- **面试可讲** — "我实现了用户自配 LLM 的方案，用 Fernet 加密存储，体现安全意识和成本控制"

## 五、RAG（检索增强生成）

### 是什么
RAG = Retrieval-Augmented Generation。让 LLM 在生成回答前先从一个外部知识库检索相关文档，把检索结果作为上下文喂给 LLM。解决 LLM 的知识截止和幻觉问题。

### 在项目中的应用 — 面经库
```
上传面经文件 → 分片 → BGE 嵌入 → Qdrant 向量库
查询时: 用户问题 → BGE 嵌入 → Qdrant 向量搜索(top-20) → Cross-encoder 重排(top-5) → 喂给 LLM
```

### RAG 流水线各环节

| 步骤 | 技术 | 做什么 |
|------|------|--------|
| 分片 | 自定义 chunker | 按公司/面试轮次切割，Q&A 对保持完整；兜底滑动窗口 300-500 字符 |
| 嵌入 | BAAI/bge-small-zh-v1.5 | 文本→512维向量，中英双语，本地免费 |
| 存储 | Qdrant | 存向量+元数据(公司/岗位/轮次)，支持 metadata 过滤 |
| 召回 | 向量相似度搜索 | top-20 粗筛 |
| 重排 | cross-encoder/ms-marco-MiniLM-L-6-v2 | 对 top-20 精排到 top-5 |
| 生成 | LLM | 将 top-5 面经作为参考上下文，生成备考建议 |

### 为什么两阶段检索（粗召回 + 精排）
- **向量检索**— 速度快(O(log n))，但"语义相似≠最相关"
- **Cross-encoder** — 对每个(query, doc)对做交叉注意力，精度高但慢
- **组合** — 先粗筛 20 条，再精排 5 条，兼顾速度和精度

### 为什么选 BGE-small-zh-v1.5
- 中英双语，中英混杂 JD/简历场景优于纯英文模型
- 512 维，轻量，本地运行，零 API 费用
- 替代方案 OpenAI text-embedding-3-small 按 token 收费

## 六、向量数据库 — Qdrant

### 是什么
专用的向量搜索引擎。存高维向量 + 元数据，支持 ANN（近似最近邻）搜索和元数据过滤。

### 在项目中的应用
- 存面经 chunks 的 512 维向量 + 元数据(公司/岗位/轮次/日期)
- 检索时先按公司/岗位过滤，再向量搜索，再送给 cross-encoder 重排

### 为什么选 Qdrant
- **Payload 过滤** — 支持 `must` 条件过滤后再检索（先筛选公司再搜），ChromaDB 做不到
- **Docker 一键部署**
- **Rust 实现**，性能高

## 七、流式输出 — SSE (Server-Sent Events)

### 是什么
HTTP 协议上的单向推送机制。服务端持续发送 `event: xxx\ndata: {...}\n\n` 格式的数据流，前端用 `EventSource` 或 `fetch` + `ReadableStream` 逐条消费。

### 在项目中的应用
- 分析匹配结果逐 token 流式输出到前端
- 面试对话逐轮评分 + 追问流式返回
- 公司调研结果流式生成
- 前端统一用 `lib/sse.ts` 的 `readSSE()` + `fetchSSE()` 消费流

### 为什么选 SSE
- 场景是单向推送（服务器→客户端），SSE 完全满足
- WebSocket 是双向协议，引入不必要的握手和保活复杂度
- 比 WebSocket 更轻量，浏览器原生支持自动重连

## 八、分层 Memory（模拟面试）

### 是什么
在长对话中管理 LLM 的上下文窗口。不把所有历史塞进 prompt（会超 token 限制），而是分层存储。

### 在项目中的应用 — 三层结构

| 层 | 存储内容 | 策略 |
|------|---------|------|
| 短期 | 最近 3 轮原文 | 滑动窗口，超量移入长期层 |
| 长期 | 历史对话摘要 | LLM 压缩，存为一段文字 |
| 结构化 | 关键实体（项目/技能/经验） | 每轮提取 JSON 实体，追加更新 |

### 为什么这样设计
- **全量历史超 context 窗口** — LLM 通常 4K-128K tokens，20+ 轮面试会爆
- **分层策略** — 重要信息用结构化存储（精确），次要信息用摘要（节省空间），最近对话保留原文（连贯性）

## 九、文档解析

### 是什么
从 Word (.docx) 和 PDF 文件中提取纯文本。

### 在项目中的应用
- **python-docx** — 主解析器。读 .docx 段落+表格。对 WPS 创建的文档，
  python-docx 的段落 API 可能返回空，此时 fallback 到原始 XML 提取所有 `w:t` 文本节点
- **PyMuPDF** — PDF 解析。用 `fitz.open()` + `page.get_text()` 逐页提取
- 格式自动判断分流，对上层 Agent 节点透明

### 为什么两种都要
- 90%+ 简历是 Word 格式，python-docx 能保留段落结构
- PDF 作为兼容方案，PyMuPDF 比 pdfplumber 更快

## 十、前端技术

### Next.js 16 (App Router)
- React 全栈框架，服务端渲染 + 客户端组件混合
- 本项目 7 个页面，`"use client"` 标记交互组件
- `lib/sse.ts` 封装 SSE 流读取

### Tailwind CSS v4
- Utility-first CSS 框架
- CSS 变量统一管理 RawBlock 设计令牌（颜色、字体、边框、圆角）

### RawBlock 设计系统
- 粗野主义风格——纯黑白、零圆角、粗边框、无阴影
- 字体：Archivo Black（标题）+ Work Sans（正文）+ Space Mono（等宽）+ Noto Sans SC（中文）
- 自定义组件库：Button/Card/Input/Chip/Tooltip 等

## 十一、认证与加密

| 技术 | 用途 |
|------|------|
| JWT (python-jose) | 用户登录后签发 token，所有 API 通过 `Authorization: Bearer` 鉴权 |
| bcrypt | 密码哈希，不存明文 |
| Fernet (cryptography) | 用户自配的 LLM API Key 加密存 PostgreSQL |

## 十二、数据库 — PostgreSQL + Docker Compose

- `users` 表 — 邮箱、密码哈希、API Key 密文、简历文本
- `interview_sessions` 表 — 面试会话、分层记忆 JSON、匹配分数
- pgvector 扩展 — 未来可做向量检索的备选

Docker Compose 一键启动 postgres + qdrant + redis。

## 十三、面试重点（按重要程度排序）

1. **LangGraph 条件路由** — 图结构 vs Chain，状态管理，事件流 — 最核心
2. **RAG 流水线** — 分片→嵌入→Qdrant→重排→生成，两阶段检索的理由
3. **分层 Memory** — 为什么需要、三层怎么配合、解决什么问题
4. **用户自配 LLM** — 加密存储、兼容 OpenAI 接口、成本控制
5. **SSE vs WebSocket** — 场景匹配、复杂度对比
6. **向量库选型** — Qdrant vs ChromaDB，payload 过滤是关键
